# Office作業のAI移譲 実装方針

社内環境（GitHub Copilot + VS Code / エンプラ標準 MS Office のみ）で、Excel・Word・Outlook 作業を最小限・軽量に AI へ移譲していくための方針メモ。

---

## 基本コンセプト

**「契約（何を）」と「変換（どう）」を分離する。**

- 入力フォーマットを **JSON** で固定（＝契約 / Input Contract）
- 実処理は **Python / PowerShell** が JSON を受け取って Office 成果物に変換（＝処理）
- これにより、AI に書かせるのは「変換ロジック」だけで済み、入出力の形は人間側が握れる

> 入出力を固定すれば、その間の処理は安全に AI へ委譲できる、という input/output コントラクトの考え方をそのまま適用。

---

## 領域ごとのツール選定

各ツールの素性に沿って言語を割り当てる。言語が2つに割れるのは恣意的分割ではなく、最適配置の結果。

| 領域 | 性質 | 採用 | 追加インストール |
|---|---|---|---|
| Excel 生成 | オフライン文書生成 | **Python（openpyxl / pandas）** | なし（Python導入済） |
| Word 生成 | オフライン文書生成 | **Python（python-docx） / 文章主体は md→Pandoc** | なし（Pandocは将来必要時） |
| Outlook 操作 | ライブ COM のグルー | **PowerShell（.ps1, 据え置き）** | なし |

### なぜ Excel / Word は Python か
- Office 本体なしでファイル形式を直接生成できる（COM のゾンビプロセス・モーダルダイアログを回避）
- `openpyxl` / `python-docx` / `pandas` は **Copilot の生成精度が最も高い**領域。AI 移譲との相性が良い
- `json` 標準モジュールで契約をネイティブに処理できる
- Python ランタイムは導入済みのため、追加ハードルなし

### なぜ Outlook は PowerShell 据え置きか
- Outlook はファイル形式ではなく**ライブの MAPI ストア**。オフラインで叩けるライブラリは存在しない
- PowerShell でも Python でも、結局 `Outlook.Application` の COM を経由する**同じ土俵**
- → Copilot の生成精度差もほぼ出ない。むしろ pywin32 は定数・バインディングでハマりどころがある
- **既に稼働している .ps1（アイテム/フォルダ一覧、アイテム移動、メールのテキスト出力、添付DL）を書き直す実利が小さい**

## 不採用とした選択肢

| 選択肢 | 不採用理由 |
|---|---|
| PowerShell + ImportExcel | Excel/Word は Copilot 精度で Python が優位。Python 導入済みのため標準同梱の利点が効かない |
| Open XML SDK | 基本機能しか使わない前提に対して冗長 |
| Office COM 自動化（常用） | 大量生成・並列に弱く後始末必須。常用はしない（※下記フォールバック時のみ） |
| Microsoft Graph API | Azure AD アプリ登録（管理者承認）が必要で採用コスト高。MCP と同種のハードルのため見送り |
| MCP 導入 | 現時点で導入ハードルあり、未採用 |

## 機能の壁に当たった時の二段構え

基本機能（セル値・書式・数式文字列・簡単なグラフ / 見出し・段落・表・画像挿入）は Python で過不足なし。
ただし純 Python・Office 非依存ゆえの弱点があり、ここに当たった機能**だけ** COM 自動化を併用する。

**普段は Python、無理な時だけ COM。**

### 想定される壁
- `openpyxl`：既存ファイルを編集保存すると**グラフ/ピボットが壊れる・消える**ことがある。数式は文字列を書くだけで**計算結果は持たない**（開くまで空）
- `python-docx`：**PDF変換・複雑なレイアウト・変更履歴**が苦手
- 共通：凝った装飾のテンプレを「開いて穴埋め保存」で装飾を保てない場合がある

→ 壁に当たって初めて、その機能に限定して COM を検討する。先回りで作り込まない。

## 当面の進め方

1. 動いている Outlook 系 .ps1 は**そのまま据え置き**
2. 新規の **Excel / Word 生成から Python を追加**
3. まず JSON スキーマ＋ `openpyxl` の最小雛形（JSON読み込み → xlsx 生成）から着手
4. 基本機能で回らない箇所が出たら、その機能単位で COM 併用を判断
5. 将来「全パイプラインを Python 1本に統一したい」と感じた場合のみ、技術優位ではなく**統一目的**で .ps1 の移植を検討

## テンプレート方針

**装飾はコードから追い出し、テンプレファイルに持たせる。** 3層に分離する。

- **装飾** → テンプレ（`.xlsx` / `.docx`）
- **データ** → JSON
- **結線** → Python

### テンプレ形式は通常の .xlsx / .docx を使う（.xltx / .dotx は不要）

- `.xltx` / `.dotx` の唯一の利点は「ダブルクリックで新規文書として開きテンプレ本体を上書きしない」という**人間のGUI操作専用の事故防止機構**
- 自動生成では Python が `load_workbook("template.xlsx")` で読み込み → **別名で `save()`** するため、上書き防止はコード側が既に担保。テンプレ形式の利点が無意味になる
- むしろ `openpyxl` は `.xltx` の扱いに難があり、ライブラリのサンプルや Copilot の学習例も `.xlsx` / `.docx` が圧倒的多数 → **AI生成精度の面でも通常形式が有利**
- 「過去ドキュメントをコピーしてテンプレ運用」という従来のやり方を、コードが自動でやっているだけ。概念的に同じ

### openpyxl のグラフ破壊問題の回避策にもなる

グラフ・数式・書式を**テンプレ側に作り込み**、Python はグラフが参照するデータ範囲のセル値だけを書く。これなら既存グラフ/ピボットを壊さない。

### ディレクトリ構成

```
templates/
  invoice_template.xlsx   ← 装飾・数式・グラフを作り込んだ「素」のxlsx
  report_template.docx    ← スタイル・ヘッダ/フッタを作り込んだdocx
outputs/
  invoice_2026-06-07.xlsx ← Pythonが穴埋めして別名保存
```

### 運用ルール

- 保存先は必ず `outputs/` に固定。テンプレは**読むだけ**
- 心配なら `templates/` を Git 管理 or ファイルを読み取り専用属性にする
- これで `.xltx` / `.dotx` の事故防止機能は完全に代替できる

## 入力契約の定義（JSON Schema + example.json）

**キー名だけでは契約として弱い。** キー名は「項目の存在」しか定義できず、「正しい値の条件」（型・単位・制約・例）を語れない。AIは値が揺れ、複数ユーザーでは解釈が割れる。

> 素のJSONはコメント非対応（`//` を書くと不正JSON）のため、example.json にコメントでfew-shotを兼ねる案は不可。

### 二本立てで定義する

1. **`schema.json`（JSON Schema）** ── 契約の本体。これ1枚で3役を兼ねる
   - `description` → AI・人間への指示（few-shot的説明）
   - `type` / `enum` / `pattern` / `required` → 機械検証
2. **`example.json`** ── 動く正解サンプル1つ（few-shot本体）。Copilotに食わせる時も人が真似る時も実例が最速

```json
// schema.json（抜粋イメージ）
{
  "properties": {
    "invoice_date": {
      "type": "string",
      "pattern": "^\\d{4}-\\d{2}-\\d{2}$",
      "description": "請求日。ISO形式 YYYY-MM-DD。例: 2026-06-07"
    },
    "amount": {
      "type": "integer",
      "description": "金額。税抜・円単位の整数。カンマ・小数なし"
    }
  },
  "required": ["invoice_date", "amount"]
}
```

### 検証

Python の `jsonschema` ライブラリで、**Office生成に流す前に入力を検証**して不正入力を弾く。複数ユーザー運用で効く。

### 入力主体による使い分け

- **AI中心** → JSON Schema + example.json（推奨）
- **人間中心で書きながら説明を見たい** → 入力形式をコメント付きYAMLにする手もある。ただしコメントは実行時に消え仕様の保管場所にはならないため、**別途 schema.json で検証**する
- いずれの場合も**検証用 schema.json は必ず持つ**
