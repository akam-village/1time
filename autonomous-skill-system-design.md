# 自律型スキルシステム 設計思想

**核心：スキルを増やすことより、スキルを制御するインフラを先に作る**

---

## 1. 問いの整理

> 「スキル増やしすぎると良くない」は正しい。でも問題はスキルの数ではなく、**オーケストレーターがどうスキルを選ぶか**の仕組みがないこと。

| 増殖の症状 | 根本原因 |
|---|---|
| どのスキルを呼ぶか毎回迷う | スキルの目次（INDEX）がない |
| 似たスキルが乱立する | 統廃合の判断基準がない |
| 追加すべきか判断できない | タスク頻度・所要時間の記録がない |
| スキル間で矛盾が生じる | 共通ルール（ガバナンス）が薄い |

解決策は「スキルを減らす」ではなく、**レジストリとタスクDBで成長を制御する**。

## 3. 制御の要：INDEX.md ＋ TASK-DB.json

### INDEX.md（スキルの目次）

段階的開示の核心。オーケストレーターはここだけ読んでスキルを選択する。

```markdown
# Skills Index

## Active

| スキル | トリガーKW | 優先度 | 頻度/月 | 平均所要日 |
|--------|-----------|--------|---------|-----------|
| mail   | メール/返信/受信トレイ | ★★★ | 22 | 0.1 |
| project| 進捗/締切/ステータス   | ★★☆ | 8  | 0.5 |
| report | 報告/集計/まとめ       | ★☆☆ | 3  | 1.0 |

## Deprecated
→ SKILL-DB.md 参照

## 追加閾値
頻度 ≥ 5回/月 かつ スキル未配備 → 新規スキル作成を検討
```

### TASK-DB.json（優先度算出の源泉）

```json
{
  "tasks": [
    {
      "task_name": "取引先への返信メール作成",
      "frequency_per_month": 22,
      "avg_days": 0.1,
      "last_run": "2026-06-10",
      "skill_assigned": "mail",
      "notes": "定型文あり→高度に自動化可"
    },
    {
      "task_name": "週次進捗レポート",
      "frequency_per_month": 4,
      "avg_days": 0.5,
      "last_run": "2026-06-09",
      "skill_assigned": "project",
      "notes": "データ取得がボトルネック"
    },
    {
      "task_name": "UL認証書類整理",
      "frequency_per_month": 1,
      "avg_days": 3.0,
      "last_run": "2026-05-20",
      "skill_assigned": null,
      "notes": "頻度低・スキル化は後回し"
    }

}
```

**棚卸しの手順：**
1. 過去3ヶ月の着手タスクをすべて列挙
2. 頻度 × 所要日数でスコア算出（インパクト = 頻度 × 平均日数）
3. インパクト上位から順にスキル配備
4. 頻度 < 2回/月 のものはスキル化しない（手動で十分）

## 4. スキル設計の3原則

### 原則1：スキルは厚く、エージェントは薄く

```
悪い例：
  manager.agent.md に「メールの返信は丁寧な敬語で…」と書く

良い例：
  mail/SKILL.md に入出力規約と参照先を書く
  mail/references/tone-guide.md に敬語ルールを書く
  manager.agent.md は「mail SKILLに委譲する」だけ書く
```

### 原則2：references/ で段階的開示

```
コンテキスト消費の3層構造：

L1: INDEX.md のスキル名・トリガーKWのみ   ← 常時（軽い）
L2: SKILL.md 全体                          ← スキル発動時
L3: references/ の各ファイル               ← SKILL.mdから参照時のみ

10〜20のスキルを登録しても、
実際に展開されるのは同時に1〜3程度。
```

### 原則3：矛盾防止は入出力契約で担保

各SKILL.mdに必ず書く項目：

```markdown
## 入力契約
- task_type: "reply" | "new" | "forward"
- recipient_info: { name, relationship, language }
- content_points: string[]

## 出力契約
- subject: string（20字以内）
- body: string（敬体・です/ます）
- confidence: 0.0〜1.0

## 禁止事項
- 添付ファイルの生成
- 外部APIへの直接呼び出し（Scriptに委譲すること）
```

入出力が固定されていれば、スキルが増えても**他スキルとの干渉は原理的に起きない**。

## 5. 自律性のリアルな上限

> 「1ワークスペースでメール処理〜プロジェクト進捗管理〜派生タスクを全部こなす」

これは実現可能だが、**自律の範囲を設計段階で明示しないと破綻する**。

### 自律できること / できないこと

| カテゴリ | 自律できる | 人間が必要 |
|---|---|---|
| メール | 分類・下書き・定型返信 | 最終送信の承認 |
| 進捗管理 | 状況収集・サマリ・アラート | 判断・意思決定 |
| 派生タスク | 既知パターンのタスク生成 | 新規・例外対応 |
| スキル追加 | TASK-DBへの記録 | スキル設計・配備 |

### 推奨：「自律実行」と「承認待ち」を明示する

```markdown
# manager.agent.md の一部

## 自律実行（確認不要）
- 受信メールの分類・ラベリング
- 進捗サマリの生成・保存
- TASK-DBへの実績記録

## 承認待ち（実行前に提示）
- メール送信（下書きを提示して確認）
- 新規スキルの追加提案
- スケジュールの変更・削除
```

## 6. スキル配備ロードマップ（推奨順）

```
Phase 1：基盤 ─────────────────────────
  ① TASK-DB.json の作成（3ヶ月分の棚卸し）
  ② INDEX.md の作成
  ③ manager.agent.md の作成（委譲ロジックのみ）

Phase 2：高頻度スキルから ───────────────
  ④ mail/SKILL.md（頻度最大・ROI最高）
  ⑤ project/SKILL.md（週次定常タスク）

Phase 3：中頻度スキル ──────────────────
  ⑥ report/SKILL.md
  ⑦ （TASK-DBの頻度閾値を超えたものから順に）

Phase 4：レビュー ──────────────────────
  ⑧ SKILL-DB.md で統廃合検討（3ヶ月ごと）
  ⑨ INDEX.md の優先度更新
```

---

## 7. 統廃合の判断基準（SKILL-DB.md 用）

| 統廃合パターン | 条件 | 対応 |
|---|---|---|
| Deprecate | 頻度 < 1回/月 が3ヶ月継続 | SKILL.mdをarchive/に移動 |
| Merge | 入力契約が80%以上同じ2スキル | 統合してreferences/で分岐 |
| Split | SKILL.mdが300行超 | サブスキルに分割し親SKILLから参照 |
| Rename | トリガーKWが重複 | INDEX.mdのKWを整理 |

---

## まとめ

**スキルが増えても破綻しない条件：**

1. **INDEX.mdが存在する** → オーケストレーターがスキルを迷わず選べる
2. **TASK-DBが更新される** → スキル追加の優先度判断に感情が入らない
3. **入出力契約が書かれている** → スキル間の矛盾が原理的に起きない
4. **references/を使う** → コンテキスト消費が線形に増えない
5. **自律範囲が明示されている** → 人間が介入すべき場面が明確

> 「自律性を育てる」とは、スキルを増やすことではなく、  
> **スキルが自己組織化できるインフラを作ること**。
